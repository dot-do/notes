## Payload CMS Collections (4)

### 1. Apis Collection
**Main API directory entries**

**Fields:**
- `name` - API name (e.g., "OpenWeatherMap")
- `slug` - URL-friendly identifier (e.g., "openweathermap")
- `description` - What the API does
- `baseURL` - API endpoint (e.g., "https://api.openweathermap.org/data/2.5")
- `category` - REST, GraphQL, SOAP, WebSocket, gRPC, Webhook
- `pricing` - Free, Freemium, Paid, Enterprise
- `authType` - API Key, OAuth, JWT, Basic, None
- `documentation` - Docs URL
- `provider` - Company/organization providing the API
- `website` - Provider website
- `logo` - Logo image URL
- `rateLimits` - Requests per hour/day/month
- `uptime` - Uptime percentage (auto-calculated)
- `averageResponseTime` - Response time in ms (auto-calculated)
- `supportedRegions` - NA, EU, APAC, SA, AF, Global
- `features` - Array of feature strings
- `tags` - Array of search tags
- `status` - Active, Deprecated, Sunset
- `verified` - Verified by directory.ninja team

**Access Control:**
- Read: Public
- Create/Update/Delete: Authenticated users only

### 2. Categories Collection
**API categorization and organization**

**Fields:**
- `name` - Category name (e.g., "Weather APIs")
- `slug` - URL-friendly identifier
- `description` - Category description
- `icon` - Icon identifier or emoji
- `count` - Number of APIs in category (auto-calculated)
- `parent` - Parent category for nested structure
- `featured` - Show in featured categories

**Access Control:**
- Read: Public
- Create/Update/Delete: Authenticated users only

### 3. Uptime Collection
**Historical uptime monitoring data**

**Fields:**
- `api` - Relationship to Apis collection
- `timestamp` - When the check occurred
- `status` - Up, Down, Degraded, Timeout
- `responseTime` - Response time in milliseconds
- `statusCode` - HTTP status code (e.g., 200, 404, 500)
- `region` - US East, US West, EU, APAC
- `errorMessage` - Error details if status is down/degraded

**Access Control:**
- Read: Public
- Create: Uptime monitor worker
- Update/Delete: Authenticated users only

### 4. Reviews Collection
**User-submitted API reviews and ratings**

**Fields:**
- `api` - Relationship to Apis collection
- `userId` - User ID from auth system
- `title` - Review title
- `rating` - 1-5 stars
- `comment` - Review text
- `pros` - Array of pros
- `cons` - Array of cons
- `useCaseDescription` - How the reviewer uses this API
- `verified` - Verified as legitimate review
- `helpful` - Number of helpful votes

**Access Control:**
- Read: Public
- Create: Authenticated users
- Update/Delete: Author or admin only

## MCP Server Implementation

**5 Tools for AI Agents:**

### 1. `search_apis`
Search and filter the API directory

**Input:**
- `query` (string, optional) - Search query
- `category` (enum, optional) - Filter by category
- `pricing` (enum, optional) - Filter by pricing
- `authType` (enum, optional) - Filter by auth type
- `tags` (array, optional) - Filter by tags
- `limit` (number, default: 10) - Results to return
- `offset` (number, default: 0) - Pagination offset

**Output:** List of matching APIs with all fields

### 2. `check_uptime`
Get uptime statistics and monitoring data

**Input:**
- `apiId` (string) - API ID to check
- `days` (number, default: 7) - Days to analyze

**Output:** Uptime percentage, average response time, incidents

### 3. `compare_apis`
Compare multiple APIs side-by-side

**Input:**
- `apiIds` (array, 2-5 IDs) - APIs to compare

**Output:** Comparison of features, pricing, uptime, reviews

### 4. `get_api_details`
Get comprehensive details about a specific API

**Input:**
- `apiId` (string) - API ID

**Output:** Full API details, documentation, rate limits, reviews

### 5. `submit_review`
Submit a review for an API

**Input:**
- `apiId` (string) - API to review
- `userId` (string) - User submitting review
- `title` (string) - Review title
- `rating` (number, 1-5) - Star rating
- `comment` (string) - Review text
- `pros` (array, optional) - List of pros
- `cons` (array, optional) - List of cons
- `useCase` (string, optional) - Use case description

**Output:** Review ID and success status

**3 Resources:**

1. `apis://all` - Complete list of all APIs
2. `apis://categories` - List of categories with counts
3. `apis://uptime/{id}` - Historical uptime data for specific API

## Uptime Monitoring

### Implementation

**Worker**: `src/uptime.ts`
**Schedule**: Every 15 minutes (Cloudflare Workers cron)
**Regions**: US East, US West, EU, APAC

### How It Works

1. **Fetch Active APIs** - Query database for all `status='active'` APIs
2. **Multi-Region Checks** - Check each API from 4 regions
3. **Health Metrics** - Measure response time, status code
4. **Store Results** - Insert into `uptime` table
5. **Aggregate Stats** - Calculate 7-day uptime percentage and avg response time
6. **Update API Records** - Update `uptime` and `averageResponseTime` fields

### Status Types

- ✅ **Up** - HTTP 2xx response
- ⚠️ **Degraded** - HTTP 4xx response
- ❌ **Down** - HTTP 5xx or network error
- ⏱️ **Timeout** - Request exceeds 10 seconds

### Features

- **Automatic Retries** - None (single attempt per region per check)
- **Timeout Handling** - 10 second timeout per request
- **Error Logging** - Stores error messages in database
- **Manual Trigger** - HTTP endpoint for on-demand checks (requires secret)

### Cron Configuration

```json
"triggers": {
  "crons": ["*/15 * * * *"]
}
```

**Frequency**: Every 15 minutes = 96 checks per day per API
**Total Daily Checks**: 96 checks × 4 regions = 384 checks per API per day
