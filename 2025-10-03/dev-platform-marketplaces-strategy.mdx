# Developer Platform Marketplaces Strategy

**Date:** 2025-10-03
**Status:** Research Complete
**Purpose:** Comprehensive marketplace strategy across GitHub, Cloudflare, Vercel, and Netlify

---

## Executive Summary

This document outlines opportunities and strategies for publishing integrations, tools, and services across major developer platforms in 2025. The landscape is shifting toward:

- **MCP (Model Context Protocol)** replacing GitHub Copilot Extensions (Nov 2025)
- **CLI-first integrations** on Cloudflare (moved from GUI marketplace in June 2025)
- **OAuth2-based native integrations** on Vercel and Netlify
- **95% revenue share** on GitHub Marketplace (up from 75%)
- **Federated registry systems** for AI agent tools via MCP

**Key Insight:** The developer marketplace landscape is consolidating around standardized protocols (MCP, OAuth2, OpenAPI) while individual platforms maintain unique value propositions for framework optimization, edge computing, and developer experience.

## 1. GitHub Marketplace

### 1.1 GitHub Actions Publishing

**Overview:**
GitHub Marketplace hosts 10,000+ actions with immediate publication (no review required). Actions are the most accessible entry point for developer tools.

**Requirements:**
- Public repository with single `action.yml` or `action.yaml` at root
- Unique action name (cannot conflict with existing marketplace listings)
- No workflow files in the repository
- Must use two-factor authentication
- Each repository = one action (no multi-action repos)

**Publishing Process:**
1. Create `action.yml` metadata file in repository root
2. Navigate to action metadata file in GitHub UI
3. Click "Draft a release" banner
4. Select "Publish this Action to the GitHub Marketplace"
5. Action goes live immediately (no approval delay)

**Categories (2025):**
- **Code Quality** - Linting, formatting, code review automation
- **CI/CD** - Build, test, deployment automation
- **Security** - Vulnerability scanning, secret detection
- **Monitoring** - Performance tracking, error reporting
- **Sustainability** (NEW Feb 2025) - Environmental impact optimization
- **Project Management** - Issue triage, automation
- **Code Review** - PR automation, review workflows
- **Testing** - Test automation, coverage reporting

**Discoverability Strategy:**
```yaml
# action.yml - Optimize for discovery
name: 'Short Descriptive Name'
description: 'Clear value proposition with keywords'
author: 'Your Organization'
branding:
  icon: 'relevant-feather-icon'
  color: 'brand-color'
inputs:
  # Document all inputs clearly
runs:
  using: 'node20'
  main: 'dist/index.js'
```

**Best Practices:**
- **Name Strategy:** Use keywords users search for (e.g., "Deploy to Cloudflare Workers")
- **Documentation:** Comprehensive README with usage examples
- **Versioning:** Use semantic versioning and GitHub releases
- **Testing:** Include workflow examples in `.github/workflows/`
- **Branding:** Choose distinctive icon and color for visual recognition

**Monetization:**
- GitHub Actions are **always free** (cannot charge for Actions)
- Monetization via GitHub Apps (see 1.2)

### 1.2 GitHub Apps Publishing

**Overview:**
GitHub Apps can be free or paid, with apps owned by organizations eligible to sell. Significantly improved revenue share (95% vs 75% previously).

**Pricing Models:**
1. **Free Plans** - Completely free, no charges allowed
2. **Flat Rate** - Set monthly/yearly fee
3. **Per-Unit** - Price per user in organization

**Pricing Requirements:**
- Paid plans require verified publisher status
- Prices in US dollars only
- Must offer both monthly and yearly billing
- Maximum 10 pricing plans per app
- Free 14-day trials available (auto-converts to paid)

**Revenue Share:**
- **95% to developer** (GitHub takes 5%)
- Payment processing handled by GitHub
- Monthly payouts via Stripe

**Publishing Requirements:**
- Organization-owned (for paid apps)
- Verified publisher status
- Webhook endpoint for event processing
- OAuth2 authentication flow
- Privacy policy and terms of service
- Support contact information

**Review Process:**
- Apps submitted for approval
- Security review for paid apps
- Can take 2-5 business days

**Best Practices:**
- Start with free tier to build user base
- Use free trials to convert users
- Implement webhook reliability (retry logic)
- Provide excellent documentation
- Offer responsive support

### 1.4 GitHub Copilot Extensions (DEPRECATED)

**⚠️ SUNSET NOTICE:** GitHub App-based Copilot Extensions will be **disabled November 10, 2025** in favor of MCP servers.

**Timeline:**
- **September 24, 2025:** New server-side Copilot Extensions blocked
- **November 3-7, 2025:** Brownout testing (temporary interruptions)
- **November 10, 2025, 11:59 PM PST:** Full sunset - all extensions disabled

**Migration Path:**
- Replace with **MCP (Model Context Protocol) servers**
- Architectural difference: replacement, not migration
- VS Code Copilot Extensions NOT affected (remain supported)

**Why MCP?**
- Universal standard for AI agent integration
- Build once, use across any MCP-compatible host
- Works with GitHub Copilot, Claude Code, and other agents
- Better ecosystem compatibility

**Action Required:**
- Migrate existing Copilot Extensions to MCP by Nov 10, 2025
- See Section 6 for MCP publishing strategy

**Available Endpoints:**


### 2.2 Database Integrations

**Historical Context:**
Launched May 2023 with OAuth2 flow for automatic credential management. Now CLI-based.

**Original Features (deprecated):**
- OAuth2 flow to database provider
- Automatic environment variable injection
- Encrypted credential storage
- One-click setup

**Current Approach (2025):**
```typescript
// workers/db-service/src/index.ts
export default {
  async fetch(request: Request, env: Env) {
    // Access secrets as environment variables
    const db = await connect(env.DATABASE_URL)
    // ...
  }
}
```

**Connection Methods:**

**1. HTTP-based drivers (Recommended):**
```typescript
// Neon HTTP driver
import { neon } from '@neondatabase/serverless'
const sql = neon(env.DATABASE_URL)
```

**2. TCP with Hyperdrive:**
```typescript
// Traditional TCP via Hyperdrive connection pooling
import { Client } from 'pg'
const client = new Client({ connectionString: env.HYPERDRIVE_URL })
```

**3. Native WebSockets:**
```typescript
// PlanetScale WebSocket driver
import { connect } from '@planetscale/database'
const db = connect({ url: env.DATABASE_URL })
```

**Integration Recommendations:**
- Provide HTTP-based drivers for edge compatibility
- Document Hyperdrive setup for TCP connections
- Offer connection pooling best practices
- Include latency optimization guides

### 4.2 Build Plugins Development

**Overview:**
Run JavaScript/TypeScript code at specific lifecycle events during build-deploy process.

**Plugin Lifecycle Events:**
```typescript
// netlify-plugin-example/index.ts

export const onPreBuild = async ({ utils }) => {
  console.log('Running before build command')
  // Pre-build logic
}

export const onBuild = async ({ utils }) => {
  console.log('Running during build')
  // Build-time logic
}

export const onPostBuild = async ({ utils }) => {
  console.log('Running after build completes')
  // Post-build logic
}

export const onSuccess = async ({ utils }) => {
  console.log('Build succeeded')
  // Success handler
}

export const onError = async ({ utils }) => {
  console.log('Build failed')
  // Error handler
}
```

**Additional Lifecycle Events:**
- `onPreDev` - Before dev server starts
- `onPreBuild` - Before build command
- `onBuild` - During build
- `onPostBuild` - After build
- `onSuccess` - Build succeeded
- `onError` - Build failed
- `onEnd` - Always runs at end

**Plugin Installation Methods:**

**1. UI Installation (Simple):**
- Click "Install" in Netlify UI
- Runs in all deploy contexts
- Limited configuration options

**2. File-based (`netlify.toml`):**
```toml
# netlify.toml
[[plugins]]
  package = "netlify-plugin-example"

  [plugins.inputs]
    apiKey = "abc123"
    endpoint = "https://api.example.com"

# Context-specific configuration
[context.production.plugins]
  [[context.production.plugins]]
    package = "netlify-plugin-analytics"

[context.deploy-preview.plugins]
  [[context.deploy-preview.plugins]]
    package = "netlify-plugin-lighthouse"
```

**3. Local Plugin (in repo):**
```toml
[[plugins]]
  package = "./plugins/custom-plugin"
```

**Plugin Structure:**
```
netlify-plugin-example/
├── manifest.yml       # Plugin metadata
├── index.ts          # Plugin logic
├── package.json
└── README.md
```

**manifest.yml:**
```yaml
name: netlify-plugin-example
description: Example build plugin
inputs:
  - name: apiKey
    description: API key for service
    required: true
  - name: endpoint
    description: API endpoint URL
    default: https://api.example.com
```

**Utils API:**
```typescript
export const onPostBuild = async ({ utils, inputs, constants }) => {
  // Show user message
  utils.status.show({ summary: 'Plugin complete!' })

  // Fail build
  utils.build.failBuild('Error message')

  // Cancel build
  utils.build.cancelBuild('Cancellation reason')

  // Cache files
  await utils.cache.save('/path/to/cache')
  await utils.cache.restore('/path/to/cache')

  // Access inputs
  console.log(inputs.apiKey)

  // Access constants
  console.log(constants.SITE_ID)
  console.log(constants.BUILD_ID)
}
```

## 6. MCP (Model Context Protocol) Registry

### 6.1 Overview

**What is MCP?**
Model Context Protocol is an open standard for connecting AI assistants to external data sources and tools. Launched by Anthropic in late 2024, it enables universal AI agent integration.

**Key Benefits:**
- **Build once, use everywhere** - Works with Claude, GitHub Copilot, and any MCP-compatible host
- **Standardized protocol** - JSON-RPC 2.0 based, well-defined spec
- **Open source** - Community-driven, federated ecosystem
- **Replaces fragmented ecosystems** - Single integration vs platform-specific extensions

**MCP Registry Launch:**
- **Preview:** September 8, 2025
- **Official Registry:** https://registry.modelcontextprotocol.io
- **GitHub Registry:** Curated directory backed by GitHub repos
- **Ecosystem Size:** ~16,000 servers (as of Oct 2025)

### 6.2 Publishing to MCP Registry

**Registry Architecture:**
- **Federated system** - Central registry + private sub-registries
- **OpenAPI spec** - Machine-readable catalog
- **GitHub-backed** - Each server linked to GitHub repository
- **Community-driven** - Open source contributions

**Publishing Process:**

**1. Build MCP Server:**
```typescript
// server.ts - Basic MCP server
import { Server } from '@modelcontextprotocol/sdk/server/index.js'
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js'

const server = new Server(
  {
    name: 'my-mcp-server',
    version: '1.0.0',
  },
  {
    capabilities: {
      tools: {},
    },
  }
)

// Register tools
server.setRequestHandler(ListToolsRequestSchema, async () => {
  return {
    tools: [
      {
        name: 'get_data',
        description: 'Fetch data from API',
        inputSchema: {
          type: 'object',
          properties: {
            query: { type: 'string' }
          }
        }
      }
    ]
  }
})

server.setRequestHandler(CallToolRequestSchema, async (request) => {
  const { name, arguments: args } = request.params

  if (name === 'get_data') {
    const result = await fetchData(args.query)
    return { content: [{ type: 'text', text: JSON.stringify(result) }] }
  }

  throw new Error(`Unknown tool: ${name}`)
})

// Start server
const transport = new StdioServerTransport()
await server.connect(transport)
```

```typescript
      tools: {},
    },
  }

  return {
    tools: [
      {
        name: 'example_tool',
        description: 'Does something useful',
        inputSchema: {
          type: 'object',
          properties: {
            query: {
              type: 'string',
              description: 'The query to process'
            }
          },
          required: ['query']
        }
      }
    ]
  }
})

// Handle tool calls
server.setRequestHandler(CallToolRequestSchema, async (request) => {
  const { name, arguments: args } = request.params

  if (name === 'example_tool') {
    const result = await processQuery(args.query)
    return {
      content: [
        {
          type: 'text',
          text: JSON.stringify(result, null, 2)
        }
      ]
    }
  }

  throw new Error(`Unknown tool: ${name}`)
})
```
