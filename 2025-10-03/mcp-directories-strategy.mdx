#### Publishing Process

```bash
# Install mcp-publisher
npm install -g @modelcontextprotocol/mcp-publisher

# Initialize server.json (if not exists)
cd workers/mcp
mcp-publisher init

# Authenticate (GitHub OAuth)
mcp-publisher auth login

# Validate manifest
mcp-publisher validate

# Publish to registry
mcp-publisher publish

# Verify publication
mcp-publisher get io.do.mcp
```

#### Namespace Ownership Options

1. **GitHub OAuth** (Recommended for .do)
   - Login as `dot-do` organization
   - Publishes as `io.github.dot-do/*`
   - No additional verification needed

2. **DNS Verification** (For custom domain)
   - Add TXT record: `mcp-registry-verify=<token>`
   - Publishes as `industries.do/*`
   - Proves domain ownership

3. **HTTP Verification**
   - Host file at `https://do.industries/.well-known/mcp-registry.txt`
   - Contains verification token
   - Alternative to DNS

**Recommendation:** Use GitHub OAuth initially, add DNS verification for custom namespace later.

### Packaging Strategy

#### Option 1: Single Unified Server (Current - RECOMMENDED)

**Approach:** Package existing `workers/mcp` as one comprehensive server

**Pros:**
- ✅ Already production-ready
- ✅ 47 tools in one server
- ✅ Simplest for users (one install)
- ✅ Consistent authentication

**Cons:**
- ❌ Large surface area
- ❌ All-or-nothing deployment

**Package Name:** `@dot-do/mcp-server` or `io.do.mcp`

**Use Case:** Users who want comprehensive .do integration

#### Option 2: Modular Servers (Future)

**Approach:** Split into category-specific servers

**Packages:**
- `@dot-do/mcp-github` - GitHub tools only
- `@dot-do/mcp-memory` - Knowledge graph memory
- `@dot-do/mcp-database` - Database access tools
- `@dot-do/mcp-ai` - AI generation tools
- `@dot-do/mcp-workflow` - Workflow tools

**Pros:**
- ✅ Users install only what they need
- ✅ Smaller attack surface
- ✅ Independent versioning

**Cons:**
- ❌ More maintenance overhead
- ❌ Coordination between packages
- ❌ Multiple authentications

**Use Case:** Users with specific needs (e.g., only GitHub integration)

**Recommendation:** Start with Option 1, expand to Option 2 based on demand

## Part 4: MCP Server Best Practices

### Security Best Practices

#### 1. Authentication

```typescript
// OAuth 2.0 for user consent
export class McpServer {
  async authenticate(request: Request): Promise<User> {
    const token = request.headers.get('Authorization')?.replace('Bearer ', '')

    if (!token) {
      throw new Error('Missing authentication token')
    }

    // Validate with WorkOS, Auth0, etc.
    const user = await this.env.AUTH.validateToken(token)

    if (!user) {
      throw new Error('Invalid token')
    }

    return user
  }
}
```

**Requirements:**
- ✅ OAuth 2.0 flow for user consent
- ✅ Short-lived access tokens (1-hour max)
- ✅ Refresh token rotation
- ✅ Scope-based permissions

#### 2. Role-Based Access Control (RBAC)

```typescript
const permissions = {
  'github:read': ['user', 'admin'],
  'github:write': ['admin'],
  'db:read': ['user', 'admin'],
  'db:write': ['admin'],
  'memory:read': ['user', 'admin'],
  'memory:write': ['user', 'admin'],
}

async function checkPermission(user: User, tool: string): Promise<boolean> {
  const permission = toolPermissions[tool]
  return permissions[permission]?.includes(user.role) ?? false
}
```

```typescript

  // Remove control characters
  return input.replace(/[\x00-\x1F\x7F]/g, '')
    // Limit length
    .slice(0, 10000)
    // Escape special characters

}
```

2. **Tool Poisoning** - Compromised tools execute harmful actions
3. **Data Leakage** - Sensitive data exposed via tools
4. **Authentication Bypass** - Weak auth allows unauthorized access
5. **SSRF Attacks** - Server-side request forgery
6. **Resource Exhaustion** - DoS via expensive operations

**Mitigation Strategies:**
- ✅ OAuth 2.0 with short-lived tokens
- ✅ RBAC with principle of least privilege
- ✅ Input sanitization and validation
- ✅ Rate limiting per user/tool
- ✅ Comprehensive logging and monitoring
- ✅ Regular security audits
- ✅ Tenant isolation (multi-tenancy)
