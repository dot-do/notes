### 2.1 GitHub Awesome Lists PR Automation

**Script:** `submit-awesome-lists.ts`
**Purpose:** Automate PR submissions to GitHub awesome lists
**Time Saved:** 50-60 hours (177 PRs √ó 20 min each = 59 hours)

**Features:**
- ‚úÖ Automatic repository forking
- ‚úÖ Branch creation with proper naming
- ‚úÖ Alphabetical entry insertion
- ‚úÖ Pull request creation with detailed descriptions
- ‚úÖ Rate limiting (max 15 PRs/day)
- ‚úÖ Error handling and resume capability
- ‚úÖ Dry-run mode for testing
- ‚úÖ Automatic tracking updates

**Usage:**

```bash
# Dry run (test without submitting)
npm run submit:awesome -- --dry-run --verbose

# Submit Tier 1 lists only (15 PRs)
npm run submit:awesome -- --tier=1 --limit=15

# Submit api.do to all lists
npm run submit:awesome -- --service=api.do

# Submit specific service to specific tier with limit
npm run submit:awesome -- --service=ai.do --tier=1 --limit=10

# Real submission (CAUTION: creates actual PRs)
npm run submit:awesome -- --dry-run=false
```

**Command Line Options:**

| Option | Description | Example |
|--------|-------------|---------|
| `--service=X` | Filter by service | `--service=api.do` |
| `--tier=N` | Filter by tier (1-4) | `--tier=1` |
| `--limit=N` | Max PRs to create | `--limit=15` |
| `--dry-run` | Test mode (no PRs) | `--dry-run` |
| `--verbose` | Detailed logging | `--verbose` |

## 7. Troubleshooting

### Common Issues

**Issue 1: GitHub API Rate Limit Exceeded**

```
Error: API rate limit exceeded
```

**Solution:**
- Check rate limit: `curl -H "Authorization: token $GITHUB_TOKEN" https://api.github.com/rate_limit`
- Wait until limit resets (shown in response)
- Reduce `MAX_PRS_PER_DAY` in `.env`
- Increase `REQUEST_DELAY_MS` in `.env`

---

**Issue 2: PR Creation Fails (Fork Exists)**

```
Error: Fork already exists
```

**Solution:**
- Script handles this automatically by updating existing fork
- If issue persists, manually delete fork and retry:
  - Go to your GitHub forks
  - Delete old fork
  - Re-run script

---

**Issue 3: Screenshot Capture Fails**

```
Error: Page timeout after 30000ms
```

**Solution:**
- Ensure service URLs are correct and accessible
- Increase timeout in script:
  ```typescript
  await page.goto(url, { waitUntil: 'networkidle', timeout: 60000 })
  ```
- Check if service requires authentication
- Use local development server if services not deployed yet

---

**Issue 4: Tracking Dashboard Not Loading Data**

```
Dashboard shows "0 Total Directories"
```

**Solution:**
- Verify CSV file exists: `ls ../data/submissions-tracking.csv`
- Check CSV format (headers must match expected columns)
- Ensure web server has correct path to CSV
- Check browser console for errors (F12)

---

**Issue 5: Directory Submission Helper Crashes**

```
Error: Cannot read undefined 'name'
```

**Solution:**
- Verify tracking CSV has required columns
- Check that services configuration is complete
- Run with `--verbose` flag to see detailed errors
- Reset progress file if corrupted:
  ```bash
  rm ../data/submission-progress.json
  ```

## 8. Best Practices

### GitHub PR Submissions

**‚úÖ DO:**
- Start with dry-run to verify everything works
- Submit Tier 1 lists first (highest value)
- Respect rate limits (15 PRs/day max)
- Check PRs daily for comments/feedback
- Respond to maintainer feedback within 24 hours
- Thank maintainers for merging

**‚ùå DON'T:**
- Submit all 177 PRs in one day (spam detection!)
- Ignore dry-run warnings or errors
- Skip manual verification of generated PRs
- Forget to update tracking after manual actions
- Leave maintainer comments unanswered
- Argue with maintainers about rejections

---

### Directory Submissions

**‚úÖ DO:**
- Use helper tool for consistent data entry
- Test all UTM links before submitting
- Verify assets exist and are correct size/format
- Save confirmation emails/receipts
- Update tracking immediately after submission
- Follow up after 7-14 days if no response

**‚ùå DON'T:**
- Rush through forms (typos = rejections)
- Use same description for all directories (customize!)
- Forget UTM tracking parameters
- Submit without verifying links work
- Ignore asset requirements (logos, screenshots)
- Forget to document submission details

---

### Asset Management

**‚úÖ DO:**
- Generate all assets before starting submissions
- Use consistent naming convention
- Store in organized directory structure
- Keep originals (SVG, PSD) for future edits
- Create multiple sizes/formats as needed
- Backup asset files regularly

**‚ùå DON'T:**
- Generate assets on-the-fly during submissions
- Use inconsistent sizes or formats
- Lose original source files
- Forget to optimize images (file size)
- Use copyrighted images without permission
